# 主 Agent 验收协议

## 五层验收

1. **功能复核**：实现、测试和声明一致。
2. **语义验收**：结果与原始目标、设计、范围和非目标一致。
3. **编排复核**：任务图收敛；依赖、ownership、共享面和集成结果没有孤儿或遗漏。
4. **对抗验证**：隐藏假设、反例和失败模式被独立挑战，高风险异议已有裁决。
5. **使用证明**：真实入口接通；冷启动用户或 Agent 能用；主 Agent 自己使用满意。

## 不可下放

目标取舍、语义验收、破坏性或不可逆动作裁量、最终合并发布节奏不可下放。内部 Agent 可以提供 diff、测试、日志、截图、枚举和异议；主 Agent 必须亲自回读原始材料与真实入口。

## Claim → Evidence

Agent 报告、测试摘要和 reviewer 结论先视为 claim。成为 evidence 至少满足一项：

- 主 Agent 打开对应 diff/文件并核对关键断言。
- 主 Agent fresh 运行相关命令并看到结果。
- 主 Agent 从独立日志、截图、制品或真实入口复现。

Agent 枚举可作搜索起点，判定必须 ground-truth。未跑测试标 `未验证`；环境阻塞写明 owner 和解除条件。

## 编排完整性清单

- 所有任务节点 done、明确取消或 blocked-with-owner；没有“等回调”的孤儿。
- 所有内部 Agent 已完成、被打断或明确无需继续；没有遗留运行线。
- 每个写面有唯一 owner；最终集成 diff 已复核，不只分别复核各 Worker 结果。
- 接口、DTO、schema、配置和 migration 等跨线依赖在目标态重新验证。
- Harness 启用时，task/progress/fact/decision/relation 与源码现实一致；generated ledger 不手改。

## 使用性门槛

交付前回答：谁第一个使用、从哪个真实入口进入、旧路径是否绕过新功能、如何证明强制接线、冷启动是否顺手。功能存在但未进入关键路径时，不算完成。
